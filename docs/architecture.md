# Anima Flow — Architecture

## 設計思想

Anima Flow のアーキテクチャは **認知ループ（観測 → 内在化 → 干渉）** を実現するために設計される。

OpenClaw の設計思想（Gateway / Brain / Memory / Heartbeat / Skills）を参考にしつつ、
**Cortex（内在化）を中心に据え**、観測と干渉がそれを取り囲む構造を採用する。

```
        ┌──────────────────────────────────────┐
        │            現実世界                    │
        │  Gmail, Slack, Linear, Calendar, ...  │
        └──┬───────────────────────────────┬───┘
           │                               │
           │ 観測 (Observe)      干渉 (Intervene)
           │                               │
           ▼                               ▲
    ┌─────────────┐                 ┌──────────────┐
    │ Connectors  │                 │   Gateway     │
    │ (Query /    │                 │               │
    │  Mutation)  │                 │ Slack Bot     │
    │             │                 │ (将来:        │
    │ MCP / CLI / │                 │  Web App 等)  │
    │ Events/REST │                 │               │
    └──────┬──────┘                 └──────▲───────┘
           │                               │
           │  ┌────────────────────┐       │
           │  │  Context Space     │       │
           │  │  (文脈空間)        │       │
           │  │  スコープ定義が    │       │
           │  │  観測範囲を決定    │       │
           │  └────────┬───────────┘       │
           │           │                   │
           ▼           ▼                   │
    ┌──────────────────────────────────────┴───┐
    │                 Cortex                    │
    │          (内在化 = Internalize)           │
    │                                           │
    │  ┌────────────────────────────────────┐  │
    │  │  Memory / Knowledge Base / SSoT    │  │
    │  │  (Node + Edge Graph)               │  │
    │  └────────────────────────────────────┘  │
    │                                           │
    │  ┌────────────────────────────────────┐  │
    │  │  Context Engine                    │  │
    │  │  (想起 = 何を思い出すか)           │  │
    │  └────────────────────────────────────┘  │
    │                                           │
    │  ── 駆動エンジン ──────────────────────  │
    │  Brain        思考・判断                  │
    │  Heartbeat    自律的なループ駆動          │
    │  Skills       定型ワークフロー            │
    └──────────────────────────────────────────┘
```

## 認知ループと 6 コンポーネント

7 つのコンポーネントは、認知ループの 3 フェーズに位置づけられる:

| 認知ループ           | コンポーネント | 役割                                                       |
| -------------------- | -------------- | ---------------------------------------------------------- |
| **観測（スコープ）** | Context Space  | ユーザーが定義する観測範囲。認知ループのスコープを決定する |
| **観測**             | Connectors     | Context Space のスコープに沿って外部世界を知覚する         |
| **内在化**           | Cortex         | 観測を構造化・記憶・正規化する。**製品の本質**             |
| **内在化**           | Context Engine | Cortex から「今この瞬間に何を想起するか」を決める          |
| **内在化（駆動）**   | Brain          | Cortex を参照し、推論・判断する                            |
| **内在化（駆動）**   | Heartbeat      | ユーザー不在でも認知ループを回し続ける                     |
| **干渉**             | Gateway        | ユーザーへの FB・タスク登録・通知で現実世界に作用する      |

Skills は Brain が使うワークフロー定義であり、ループの特定パターンを定型化したもの。
Context Space は認知ループ全体を横断する概念であり、コネクタの観測範囲からコンテキストの想起まで影響する。

## 各コンポーネント

### Context Space — 文脈空間

ユーザーが定義する**情報収集と学習のスコープ**。
「何を観測し、何を記憶し、何について考えるか」の境界を明示的に設定する。

認知ループは Context Space によってスコープされた範囲内で回る。
これにより、無関係な情報のノイズを排除し、各文脈に集中した高品質な認知を実現する。

- ユーザーが明示的に作成、またはエージェントの提案をユーザーが承認
- エージェントが勝手に作成することはない
- 各 Context Space にはコネクタごとの観測スコープが定義される
- デフォルト Space（Personal）により、最小構成でも動作する

詳細は [context-space.md](./context-space.md) を参照。

### Connectors — 観測・干渉レイヤー

外部サービスとの接続を統一概念で抽象化する。
**Query（読み取り）** と **Mutation（書き込み）** が明確に分離されている。

Query は Context Space のスコープに沿って情報を取得し、
Mutation は Brain の判断に基づいて外部サービスに干渉する。

4 つの接続方式: MCP / CLI / Events / REST。
詳細は [connectors.md](./connectors.md) を参照。

### Cortex — 内在化レイヤー（中核）

Node（情報の最小単位）と Edge（Node 間の関連）で構成されるグラフ構造。
エージェントの脳の中核であり、3 つの役割を持つ:

- **Memory**: 経験・記憶（エピソード、人物認識、行動パターン）
- **Knowledge Base**: 蓄積されたナレッジ（手順書、業務知識、学び）
- **SSoT**: 散在する情報の正規化された全体像（プロジェクト状態、意思決定ログ）

Connectors から観測された情報はここに内在化され、Gateway への干渉のための文脈を提供する。
詳細は [cortex.md](./cortex.md) を参照。

### Context Engine — 想起エンジン

Cortex に蓄積された数千の Node から「今この瞬間に何を思い出すか」を決める。
内在化レイヤーの出力インターフェースであり、**製品の知的財産の中核**。
詳細は [context-engine.md](./context-engine.md) を参照。

### Brain — 駆動エンジン（思考・判断）

Claude Agent SDK をベースとした LLM エージェント。
Context Engine が想起した文脈を使い、観測の解釈・干渉の判断を行う。

### Heartbeat — 駆動エンジン（自律ループ）

定期実行ループにより、ユーザー不在でも認知ループを回し続ける。
これが Anima Flow を「待つだけのチャットボット」から「自律的な第二の脳」に変える。

- 定期的な**観測**: Inbox 監視・重要情報の検知
- 定期的な**内在化**: 記憶の定着・SSoT 同期・Knowledge Base 更新
- 定期的な**干渉**: リマインド送信・ブリーフィング・振り返り

### Gateway — 干渉レイヤー

ユーザーへの FB と、ユーザーからの入力を受け取る**双方向チャネル**。
ユーザーの修正・FB は Gateway を通じて Cortex に戻り、認知ループが閉じる。

- フェーズ 1: Claude Code（ターミナル）
- フェーズ 1.5: Slack Bot（Slack Bolt）
- フェーズ 2: カスタム Web App

### Skills — 定型ワークフロー

ユーザーが呼び出す、または Heartbeat が自動実行する定型ワークフロー。
認知ループの特定パターン（朝のブリーフィング、Inbox 処理等）を定義する。
詳細は [workflows.md](./workflows.md) を参照。

## 認知プロセスとの対応

```
認知ループ     人間の脳               Anima Flow           コンポーネント
──────────────────────────────────────────────────────────────────────
観測           注意（attention）  ←→  観測スコープの定義    Context Space
               五感（知覚）      ←→  スコープ内の情報取得  Connectors
内在化         海馬（記憶形成）   ←→  記憶の定着           Cortex (Memory)
               大脳皮質（知識）   ←→  ナレッジの蓄積       Cortex (Knowledge Base)
               前頭前皮質（判断） ←→  推論・意思決定       Brain
               ワーキングメモリ   ←→  今日のコンテキスト   Cortex (DailyNote)
               メタ認知          ←→  自己パターン認識      Cortex (Meta Memory)
               世界モデル        ←→  外界の正規化         Cortex (SSoT)
干渉           運動系（行動）    ←→  通知・タスク登録      Gateway + Heartbeat
```

## フェーズ構成

### フェーズ 1: Claude Code + MCP（開発・検証）

自分自身で使いながら認知ループと Cortex 構造を検証する。
まず観測（Connectors）と内在化（Cortex）を構築し、干渉は手動。

### フェーズ 1.5: Slack Bot + Claude Agent SDK（常駐エージェント）

Slack DM を通じた非同期対話型の常駐エージェント。
Heartbeat により認知ループが自律的に回り始める。

### フェーズ 2: スタンドアロンエージェント（製品化）

Claude Agent SDK を使い、一般ユーザー向けの Web アプリ/ボットとして提供する。
フェーズ 1-1.5 で検証した認知ループと Cortex アーキテクチャを移植する。
