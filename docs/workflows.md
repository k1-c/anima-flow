# Anima Flow — Workflows & Skills

## 概要

Anima Flow のワークフローは GTD（Getting Things Done）をベースに設計する。
日常的に使うワークフローを Skill として定義し、ユーザーが呼び出す、
または Heartbeat が自動的に実行する。

各 Skill は認知ループ（観測 → 内在化 → 干渉）の**特定パターンを定型化したもの**であり、
ループを効率的に回すための具体的な手順を定義する。

## GTD ワークフロー

### Inbox 収集（Capture）

各チャネルから未処理の情報を自動収集し、Cortex に inbox Node として起票する。

| ソース            | 収集対象                             | Connector           |
| ----------------- | ------------------------------------ | ------------------- |
| Gmail             | 未読メール・要対応メール             | MCP (Google)        |
| Slack             | メンション・DM・リアクション待ち     | Events (Slack Bolt) |
| Chatwork          | 未読メッセージ（A社様）     | REST (自作)         |
| Linear            | アサインされた Issue・期限近い Issue | MCP (Linear)        |
| Todoist           | 今日/期限切れタスク                  | MCP (Todoist)       |
| Google カレンダー | 今日/明日の予定                      | MCP (Google)        |

#### 重複防止と完了判定

Inbox 処理の核心は**「同じ情報を二重に起票しない」**ことと**「終わったものを正確に判定する」**ことにある。

##### 1. 外部 ID による重複防止（冪等性）

各 inbox Node は `source` と `external_id` の組み合わせでユニークに識別される:

```
Node: type: 'inbox'
  metadata: {
    source: 'gmail',              -- どの Connector から来たか
    external_id: 'msg_abc123',    -- 外部サービス上の一意 ID
    status: 'unprocessed',        -- unprocessed / processed / skipped
    captured_at: '2026-03-14T09:00:00Z'
  }
```

収集時に `(source, external_id)` が既存の inbox Node と一致すれば起票しない。
これにより Heartbeat が何度収集を実行しても同じ情報が重複しない（冪等）。

| ソース   | external_id の例 | 備考 |
|---------|-----------------|------|
| Gmail   | メッセージ ID (`msg_xxx`) | スレッド単位ではなくメッセージ単位 |
| Slack   | `channel_id:ts` | タイムスタンプがメッセージの一意キー |
| Linear  | Issue ID (`ISS-123`) | |
| Todoist | タスク ID | |
| Calendar | イベント ID | |
| Chatwork | メッセージ ID | |

##### 2. 完了判定（外部状態の観測）

「終わったか終わってないか」の判定は、**外部サービス上の状態**と **Cortex 上の処理状態**の両方を見る:

```
                  外部サービスの状態（観測）
                  ┌─────────────────────────────────┐
                  │ Gmail: 返信済み / ラベル付与済み │
                  │ Slack: スレッド解決済み / ✅付き │
                  │ Linear: Issue がクローズ         │
                  │ Todoist: タスク完了              │
                  └──────────────┬──────────────────┘
                                 │
                                 ▼
                  ┌─────────────────────────────────┐
                  │   完了判定ロジック（Brain）      │
                  │                                  │
                  │   外部状態 + Cortex の処理状態   │
                  │   → 収集対象から除外するか判定   │
                  └──────────────┬──────────────────┘
                                 │
                  ┌──────────────┴──────────────────┐
                  │  収集対象     │   収集しない     │
                  │  (未処理)     │   (処理済み)     │
                  └─────────────────────────────────┘
```

完了判定のルール:

| ソース | 「未完了」と判定する条件 | 「完了」と判定する条件 |
|-------|------------------------|---------------------|
| Gmail | 未読 or 未返信 & 要対応フラグ | 返信済み or アーカイブ済み |
| Slack | 未応答のメンション・DM | 返信済み or リアクション済み or スレッド解決 |
| Linear | Open or In Progress & 自分アサイン | Done or Cancelled |
| Todoist | 未完了 & 期限内 or 期限切れ | 完了済み |
| Calendar | 今日/明日の予定 | 過去の予定（自動的に対象外） |
| Chatwork | 未読 | 既読 & 返信済み |

##### 3. クロスソース重複（同一トピックの名寄せ）

異なるソースから同じトピックが流入するケースがある:
- Slack で「レポートお願い」→ Gmail でも「レポートの件」
- Linear に Issue 作成 → Slack に通知

これは**起票は別々に行うが、Edge で関連づける**ことで解決する:

```
inbox Node: Slackメンション「レポートの件」
  Edge → inbox Node: Gmailメール「レポートの件」  relation: 'same_topic'
```

名寄せは Brain（LLM）が以下の手がかりで判断する:
- 時間的近接性（数時間以内の同一トピック）
- 登場人物の一致
- タイトル・内容のセマンティック類似度

名寄せされた inbox Node 群は、整理時にまとめて 1 つのアクションとして処理する。

### 整理（Clarify & Organize）

収集した情報を Brain（LLM）が分類する。
**名寄せ済みの inbox Node 群はまとめて 1 つの判断対象**として扱う:

- **即座に実行可能（2 分以内）** → 実行 or 実行を促す → inbox Node を `processed` に
- **次のアクションが明確** → タスクとして登録（Linear or Todoist）→ inbox Node を `processed` に
- **曖昧・大きい** → **タスク分割**して具体的なアクションに分解 → inbox Node を `processed` に
- **参考情報** → Cortex の Knowledge Base にアーカイブ → inbox Node を `processed` に
- **不要** → inbox Node を `skipped` に

処理された inbox Node は次回の収集時に収集対象から除外される。

### タスク分割（Breakdown）

Anima Flow の最大の差別化ポイント。
LLM が曖昧なタスクを具体的なアクションに分解する:

1. タスクのゴールを明確化する（ユーザーと対話）
2. 具体的なアクション（15-30 分単位）に分解する
3. 依存関係・優先順位を整理する
4. Linear または Todoist にサブタスクとして登録する

既存のタスク管理ツール単体ではできないことであり、
「やる気が出ない → タスクが大きく曖昧だから」という課題を直接解決する。

## Skills 設計

### `/briefing` — ブリーフィング

**トリガー**: ユーザーが呼び出す or Heartbeat が自動実行

1. 今日のカレンダー予定を取得
2. 未処理の Inbox を収集（Gmail、Slack、Linear）
3. 今日のタスク一覧を取得（Linear + Todoist）
4. Cortex から関連する記憶・文脈を想起
5. 優先順位付きのブリーフィングを生成
6. DailyNote（ワーキングメモリ）を初期化

### `/inbox` — Inbox 処理

**トリガー**: ユーザーが呼び出す or Heartbeat が定期実行

1. 全 Connector から未処理情報を収集（`(source, external_id)` で重複チェック）
2. 外部サービスの完了状態を確認し、既に終わったものは除外
3. クロスソースの名寄せ（同一トピックを Edge で関連づけ）
4. GTD フローに沿って分類を提案（名寄せ済みはまとめて判断）
5. ユーザーの承認を得てタスク登録・アーカイブ
6. 処理した inbox Node を `processed` / `skipped` にステータス更新
7. 重要な情報は Cortex に記録（エピソード Node / SSoT 更新）

### `/breakdown [タスク]` — タスク分割

**トリガー**: ユーザーが呼び出す

1. 指定タスクの内容を把握
2. Cortex から関連する知識・過去の経験を想起
3. ゴールを確認（対話）
4. 15-30 分単位のアクションに分解
5. サブタスクとして登録（Linear or Todoist）

### `/review` — 1 日の振り返り

**トリガー**: ユーザーが呼び出す or Heartbeat が毎夕自動実行

1. 今日の活動を各 Connector から収集
2. 完了 / 進行中 / 未着手を整理
3. DailyNote を完成させて Cortex に保存
4. 記憶の定着処理を実行（エピソード → 長期記憶）
5. SSoT を更新（Context Space の状態反映）
6. 明日の準備事項を提示

## Heartbeat による自動ワークフロー

Heartbeat は**ユーザー不在でも認知ループを回し続ける**駆動エンジンである。

| タイミング         | 認知ループ           | 実行内容                                                                      |
| ------------------ | -------------------- | ----------------------------------------------------------------------------- |
| 毎朝 (例: 8:00)    | 観測 → 内在化 → 干渉 | `/briefing` の自動実行。観測結果を内在化し、Gateway 経由でブリーフィングを送信 |
| 定期 (例: 30 分毎) | 観測 → 干渉          | Inbox 監視。重要な未読があれば Gateway 経由で通知                             |
| 定期 (例: 30 分毎) | 内在化 → 干渉        | リマインド。Cortex の文脈から期限が近いタスク・次の予定を事前案内             |
| 毎夕 (例: 18:00)   | 観測 → 内在化        | `/review` の自動実行。DailyNote 生成・記憶の定着処理                          |
| 深夜 (例: 3:00)    | 内在化               | Cortex メンテナンス。SSoT 同期・Knowledge Base 更新提案・Edge 整合性チェック  |
