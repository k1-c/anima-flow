# Anima Flow — Context Engine

## この文書の位置づけ

Context Engine は Anima Flow の知的財産の中核である。
認知ループの**内在化（Internalize）レイヤーの出力インターフェース**であり、
Cortex に蓄積された数千の Node から
**「今この瞬間に、どの Node を LLM に渡すか」**を決めるエンジン。

LLM のコンテキストウィンドウは有限である。
すべての Node を渡すことはできない。
かといって、重要な Node を渡し忘れれば、エージェントは的外れな干渉（行動）をする。

Context Engine は、人間の脳における**想起（recall）**のプロセスを模倣する。
**観測で集まった情報が Cortex に蓄積され、Context Engine がそこから必要な文脈を想起し、
Brain の判断を経て Gateway から干渉する** — この流れの質を決定する要。

---

## 人間の想起プロセスとの対応

認知科学における「拡散活性化モデル」（Collins & Loftus, 1975）に基づく:

```
人間の脳                        Anima Flow
──────────────────────────────────────────────────────────
1. 手がかり（cue）を受け取る     ユーザーの発話を受け取る
2. 関連する神経ノードが活性化     Cortex 内の Node が活性化
3. 活性化が隣接ノードに伝播       Edge を辿って関連 Node が活性化
4. 最も活性化したノードが意識に浮上 上位 N 件の Node がコンテキストに選出
5. 記憶が再構成される             コンテキストが組み立てられ LLM に渡る
```

## Stage 1: 手がかり抽出（Cue Extraction）

ユーザーの発話から、想起の手がかりを抽出する。
このステージ自体が LLM 呼び出し（軽量モデル）で実現できる。

### 抽出する手がかりの種類

| 手がかり | 例 | 用途 |
|---------|---|------|
| **エンティティ** | 「田中さん」「プロジェクト X」 | Stage 2 の完全一致検索 |
| **意図** | タスク確認、報告作成、質問 | コンテキストの種類を決定 |
| **時間参照** | 「先週の」「昨日の MTG」 | 時間範囲フィルタ |
| **トピック** | 「認証」「レポート」 | Stage 4 のセマンティック検索 |
| **暗黙の参照** | 「あの件」「例の話」 | 直近のエピソードから推定 |

### 「あの件」問題

「あの件どうなった？」のような曖昧な参照は、人間同士の会話では頻出する。
Context Engine はこれを解決する必要がある:

1. 直近の DailyNote（ワーキングメモリ）を参照
2. 最近のエピソードで最も活性度が高い（頻繁に触れた）話題を推定
3. 会話の直前のコンテキスト（短期的な対話履歴）から推定

## Stage 2: 直接検索（Direct Retrieval）

手がかりに対して、Cortex を直接検索する。

```sql
-- エンティティの完全一致
SELECT * FROM nodes
WHERE title ILIKE '%田中太郎%'
   OR metadata->>'aliases' ILIKE '%田中%';

-- 時間範囲フィルタ
SELECT * FROM nodes
WHERE type = 'episode'
  AND created_at BETWEEN '2026-03-07' AND '2026-03-13';

-- タイプ + メタデータフィルタ
SELECT * FROM nodes
WHERE type = 'space'
  AND metadata->>'status' = 'active';
```

このステージは**高速で確実**。完全一致する対象がある場合、ここで決着する。

---

## Stage 3: グラフ探索（Spreading Activation）

直接検索で見つかった Node を起点に、Edge グラフを辿って関連 Node を活性化する。

### 拡散活性化アルゴリズム

```
入力: 起点ノード集合 S、最大深度 D、減衰率 α

1. 起点ノードの活性化スコア = 1.0
2. for depth = 1 to D:
     for each 活性化済みノード n:
       for each リンク先ノード m:
         m.score += n.score × α^depth × edge_weight(n, m)
3. スコアが閾値以上のノードを返す
```

### 減衰率（α）の設計

Edge の種類によって伝播の強さを変える:

| Edge 種別 | 減衰率 | 理由 |
|-----------|--------|------|
| person → space | 0.8 | 人物と Context Space は強く関連 |
| space → decision | 0.9 | 意思決定は Space の重要な文脈 |
| episode → person | 0.7 | エピソードに登場した人は関連 |
| episode → episode | 0.3 | エピソード同士の関連は弱い |
| knowledge → space | 0.5 | 知識は特定 Space に紐づかないこともある |

### 例: 「田中さんに連絡して」

```
「田中さん」
  │
  ▼ (直接検索)
people/田中太郎 [score: 1.0]
  │
  ├──→ spaces/anima-flow開発 [score: 0.8]
  │      ├──→ decisions/API認証方式の決定 [score: 0.72]
  │      └──→ people/A社様 [score: 0.64]
  │
  ├──→ episodes/2026-03-13_顧客MTG [score: 0.7]
  │
  └──→ knowledge/月次レポートの手順 [score: 0.5]
```

→ 上位 5 件をコンテキストとして選出。
エージェントは「田中さん = anima-flow開発 Space のメンバーで、
昨日の顧客 MTG で要件変更があり、月次レポートも関係する人」
という豊かな文脈を持って応答できる。

## Stage 4: セマンティック検索（Semantic Retrieval）

Stage 2-3 で十分なコンテキストが得られない場合のフォールバック。
pgvector を使い、ユーザー入力のベクトルと Node の埋め込みベクトルの類似度で検索する。

```sql
SELECT *, embedding <=> $query_embedding AS distance
FROM nodes
ORDER BY distance
LIMIT 10;
```

### Stage 2-3 と Stage 4 の使い分け

| 状況 | 使うステージ |
|------|------------|
| 「田中さんについて教えて」 | Stage 2（完全一致）で十分 |
| 「認証まわりで前に何か決めたよね」 | Stage 3（グラフ探索）で decisions/ を辿る |
| 「セキュリティ的にまずいパターンってあったっけ」 | Stage 4（セマンティック検索）が必要 |

**原則: 確実な検索を先に、曖昧な検索は後に。**
ベクトル検索は便利だが、ノイズが多い。グラフ構造で辿れる関連は、グラフで辿るべき。

## Stage 5: スコアリング（Relevance Scoring）

各ステージで見つかった Node に最終スコアを付与する。
複数のシグナルを重み付き加算する:

### スコアリングファクター

```
final_score = Σ (weight_i × factor_i)
```

| ファクター | 重み（初期値） | 説明 |
|-----------|---------------|------|
| **直接一致** | 1.0 | Stage 2 で完全一致したか |
| **グラフ活性化** | 0.8 | Stage 3 での拡散活性化スコア |
| **セマンティック類似度** | 0.5 | Stage 4 でのベクトル類似度 |
| **鮮度（recency）** | 0.6 | 最終更新が新しいほど高い |
| **頻度（frequency）** | 0.4 | 過去 N 日間でアクセスされた回数 |
| **重要度（importance）** | 0.7 | metadata の priority / status による重み |
| **型ボーナス** | 可変 | 意図に応じたノート型の優遇（後述） |

### 意図に応じた型ボーナス

ユーザーの意図によって、特定の Node 型を優遇する:

| 意図 | 優遇される Node 型 |
|------|------------------|
| タスク確認 | space, daily_note |
| 人に連絡 | person, episode |
| 手順を知りたい | knowledge (procedures) |
| 振り返り | episode, daily_note |
| 意思決定 | decision, space |

### エビングハウスの忘却曲線

エピソードの鮮度は、単純な線形減衰ではなく、
忘却曲線に近い指数減衰を適用する:

```
recency_score = e^(-λ × days_since_update)
```

ただし、**繰り返しアクセスされたノートは減衰が遅くなる**（間隔反復効果）:

```
λ_effective = λ_base / (1 + log(access_count))
```

これにより:
- 一度だけ触れたエピソードは速やかに重要度が下がる
- 繰り返し参照される Node は長期記憶として定着する
- 人間の記憶の「強化」プロセスを模倣する

## Stage 6: コンテキスト組立（Context Assembly）

スコアリングされた Node を、LLM のコンテキストウィンドウに収まるよう組み立てる。

### コンテキスト予算

```
総コンテキスト予算: 128K tokens（Claude の場合）

割り当て:
├── システムプロンプト          ~2K tokens  (固定)
├── 常駐コンテキスト            ~4K tokens  (固定)
├── 想起された Node            ~8K tokens  (可変、ここが Context Engine の出力)
├── 会話履歴                   ~4K tokens  (可変)
└── ユーザー入力 + 応答余白     残り
```

### 常駐コンテキスト（Always-Present Context）

毎回必ず含めるコンテキスト。人間の「意識の背景」に相当:

| コンテキスト | 内容 | トークン目安 |
|------------|------|------------|
| **今日の DailyNote** | ワーキングメモリ。今日の予定・タスク・メモ | ~1K |
| **ユーザー設定** | type: 'preference' の Node。コミュニケーションスタイル等 | ~0.5K |
| **アクティブな Context Space 一覧** | type: 'space' で status=active の Node のタイトルとサマリ | ~1K |
| **直近の重要エピソード** | 過去 24-48h の高スコアエピソード | ~1.5K |

### 想起されたノートの配置

選出された Node は、そのまま全文を渡すとは限らない:

| Node の状態 | 処理 |
|------------|------|
| 短い（~500 tokens 以下） | 全文を渡す |
| 中程度（~2K tokens） | metadata + 要約 + 関連セクションを渡す |
| 長い（~2K tokens 超） | metadata + LLM による要約を渡す |
| Edge 先が多数 | Edge 先の Node タイトル一覧のみを渡す |

### コンテキストの構造化

渡すコンテキストは平坦なテキストではなく、構造を持たせる:

```xml
<context>
  <working_memory>
    <!-- 今日の DailyNote -->
  </working_memory>

  <recalled_notes relevance="high">
    <note type="person" id="tanaka">
      <!-- 田中太郎の全文 -->
    </note>
    <note type="space" id="anima-flow">
      <!-- プロジェクトの要約 -->
    </note>
  </recalled_notes>

  <recalled_notes relevance="medium">
    <note type="episode" id="2026-03-13-mtg">
      <!-- エピソードの要約 -->
    </note>
  </recalled_notes>

  <background>
    <!-- 進行中プロジェクト一覧 -->
    <!-- ユーザー設定 -->
  </background>
</context>
```

## 特殊なコンテキストパターン

### パターン 1: Heartbeat 実行時

ユーザーの発話がない。エージェントが自発的に行動する場面。

```
手がかり = 現在時刻 + DailyNote + 最近のエピソード

例: 朝 8:00 の Heartbeat
→ 今日の DailyNote（カレンダー予定）
→ 昨日の未完了タスク
→ 期限が今日のタスク（Linear/Todoist）
→ type: 'pattern' の Node（「月曜朝は週次レビューをする傾向」）
```

### パターン 2: 連続対話

同じスレッド内での連続対話。前の発話のコンテキストを引き継ぐ。

```
前回のコンテキスト + 新しい手がかり → マージ

前回想起: [田中太郎, anima-flow, 顧客MTG]
今回の発話: 「その要件変更の詳細は？」
→ 「その」= 前回コンテキスト内の顧客MTG
→ episodes/2026-03-13_顧客MTGでの要件変更 を高優先で追加
```

### パターン 3: 記憶の定着時

Heartbeat が記憶の定着処理を行う場面。
エージェントが「自分の記憶を整理する」ためのコンテキスト。

```
手がかり = DailyNote の全エントリ

各エントリについて:
→ 既存の Node との関連をグラフ探索
→ 新規 Node の生成が必要か判断
→ Edge の追加・更新が必要か判断
```

## メトリクスと改善

Context Engine の品質を測定するための指標:

| メトリクス | 定義 | 改善方向 |
|-----------|------|---------|
| **想起精度** | ユーザーが参照した情報がコンテキストに含まれていた割合 | 高いほど良い |
| **想起ノイズ率** | コンテキストに含めたが使われなかった Node の割合 | 低いほど良い |
| **コンテキスト利用率** | 予算のうち実際に使用した割合 | 60-80% が目標 |
| **フォールバック率** | Stage 4（セマンティック検索）まで到達した割合 | 低いほど良い（Stage 2-3 で解決できている） |
| **「あの件」解決率** | 曖昧な参照を正しく解決できた割合 | 高いほど良い |

### フィードバックループ（認知ループの閉包）

ユーザーがエージェントの応答に「違う、○○のことだよ」と修正した場合、
それは Context Engine が不適切な Node を選んだ（または選び漏れた）ことを意味する。

これは**干渉に対するユーザーの FB が Gateway を通じて Cortex に戻る**プロセスであり、
認知ループが閉じることで Context Engine 自体が改善される:

1. 修正されたケースを記録
2. どのステージで失敗したかを分析
3. スコアリングの重みを調整
4. type: 'pattern' の Node に「このパターンではこの Node が重要」と記録

**認知ループが回るほど、想起の精度が上がり、干渉の質が向上する。**

## 実装の段階

| フェーズ | 実装範囲 |
|---------|---------|
| フェーズ 1 | Stage 2（直接検索）のみ。Node 数が少ないので十分 |
| フェーズ 1.5 | Stage 3（グラフ探索）を追加。Edge の活用 |
| フェーズ 2 | 全 Stage を実装。pgvector、スコアリング、コンテキスト組立 |
| 継続的 | メトリクス計測とフィードバックループによる改善 |
