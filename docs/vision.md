# Anima Flow — Vision

## 自律的な第二の脳、もう一人の自分

Anima Flow は、単なるタスク管理ツールや AI チャットボットではない。
**自律的に知覚・記憶・思考・行動する、もう一人の自分**である。

従来の「Second Brain」（Tiago Forte の PARA メソッド等）は、
人間が手動で整理する受動的なストレージに過ぎなかった。
Anima Flow はその概念を根本から覆す。

---

## 認知ループ: 観測 → 内在化 → 干渉

Anima Flow の全体を貫く根本モデルは、3 つのフェーズからなる認知ループである。

```
        ┌──────────────────────────────────┐
        │          現実世界                 │
        └──┬───────────────────────────┬───┘
           │ 観測                  干渉 │
           ▼                          ▲
    ┌─────────────┐            ┌──────────────┐
    │ Connectors  │            │   Gateway     │
    │ (知覚)      │            │ (働きかけ)    │
    └──────┬──────┘            └──────▲───────┘
           │                          │
           ▼                          │
    ┌─────────────────────────────────┴───┐
    │              Cortex                  │
    │       (内在化 = 理解・記憶・正規化)  │
    │                                      │
    │       Brain / Heartbeat / Skills     │
    │       が駆動する                     │
    └──────────────────────────────────────┘
```

| フェーズ | 役割 | コンポーネント |
|---------|------|--------------|
| **観測（Observe）** | 外部世界を知覚し、情報を取り込む | Connectors |
| **内在化（Internalize）** | 観測を構造化・記憶・正規化する。**ここが製品の本質** | Cortex + Context Engine |
| **干渉（Intervene）** | ユーザーへの FB、タスク登録、通知など現実世界への作用 | Gateway |

Brain・Heartbeat・Skills はこの 3 フェーズを**駆動するエンジン**であり、ループを回し続ける仕組みである。

### ループの性質

- **Context Space が観測のスコープを決める** — 認知ループは Context Space（ユーザーが定義する文脈空間）の範囲内で回る。「何を観測するか」を明示的にコントロールすることで、ノイズを排除し、高品質な認知を実現する。詳細は [Context Space](./context-space.md) を参照。
- **Connectors は Cortex を育てるために存在する** — 単なる API 接続ではなく、Context Space のスコープに沿った観測が Cortex の質を決める。
- **Gateway は双方向である** — ユーザーへの干渉だけでなく、ユーザーからの FB も Gateway を通じて Cortex に戻る。ループが閉じている。
- **Heartbeat は自律的な観測サイクル** — ユーザーが何もしなくても、各 Context Space について観測→内在化→干渉が回り続ける。これが「待つだけのチャットボット」との決定的な差。

## コアバリュー: 究極のコンテキストエンジニアリング

Anima Flow の競争優位は、エージェントフレームワークでも LLM でもない。
**情報をどう構造化し、どう繋げ、今この瞬間に何を想起するか** — この設計にある。

LLM の性能は、与えるコンテキストの質で 9 割決まる。
同じモデルでも、的確なコンテキストを渡せば賢く振る舞い、
雑なコンテキストを渡せば的外れな回答をする。

認知ループの各フェーズをコンテキストエンジニアリングの視点で分解すると:

```
観測 ─────────────────────────────────────────────
0. スコープ — 何に注意を向けるか        (Context Space)
1. 収集    — スコープ内を知覚する       (Connectors)

内在化 ───────────────────────────────────────────
2. 構造化  — どう整理・分類するか       (Cortex の情報設計)
3. 接続    — 何と何を関連づけるか       (Edge = 連想グラフ)
4. 階層化  — どの記憶をどの深さに置くか  (記憶の階層)
5. 正規化  — 散在する情報をどう統合するか (SSoT)
6. 想起    — 今この瞬間に何を思い出すべきか (Context Engine)
7. 定着    — 何を覚え、何を忘れるか     (Heartbeat)

干渉 ─────────────────────────────────────────────
8. 行動    — どう働きかけるか           (Gateway)
```

**0-8 すべてが「LLM に渡すコンテキストの質を最大化する」ための設計**であり、
内在化（2-7）に 6 つのステップが集中していることが、Cortex こそが本質であることを示している。
Context Space（0）がその入口として、観測の質を決定づける。

| レイヤー | 代替可能性 | Anima Flow の位置づけ |
|---------|-----------|---------------------|
| LLM (Brain) | 高い。Claude → GPT に差し替え可能 | 利用する側 |
| Agent Framework | 高い。SDK は誰でも使える | 利用する側 |
| Gateway (Slack Bot 等) | 高い。ボイラープレート | 利用する側 |
| Context Space (注意) | 中程度。スコープ設計がS/N比を決定 | 観測の質を決める |
| Connectors (観測) | 中程度。何を観測するかの設計にノウハウ | 観測モデルの設計 |
| **情報設計 (Cortex)** | **低い。これ自体がドメイン知識** | **ここが製品** |
| **コンテキスト選択 (Context Engine)** | **低い。想起ロジックがノウハウ** | **ここが製品** |

詳細は [Context Engine 設計](./context-engine.md) を参照。

## 認知アーキテクチャとしてのエージェント

Anima Flow は人間の脳の認知プロセスを模倣した構造を持つ。
認知ループの 3 フェーズが、脳の機能領域と対応する:

```
認知ループ         人間の脳               Anima Flow
──────────────────────────────────────────────────────────
観測（Observe）    五感（知覚）      ←→  Connectors（Gmail, Slack, Calendar...）

内在化             海馬（記憶形成）   ←→  Memory（episodes → long-term への定着）
(Internalize)      大脳皮質（知識）   ←→  Cortex（Node + Edge のグラフ全体）
                   前頭前皮質（判断） ←→  Brain（Claude Agent SDK）
                   ワーキングメモリ   ←→  DailyNote（今日のコンテキスト）
                   メタ認知          ←→  Meta Memory（自分の行動パターンの認識）
                   世界モデル        ←→  SSoT（外界の正規化された理解）

干渉（Intervene）  運動系（行動）    ←→  Gateway + Heartbeat（通知、タスク登録）
```

内在化のフェーズに脳の機能が集中していることが、Cortex が本質であることを示している。

## 名前の由来

**Anima** — ラテン語で「魂・生命」。無機質なツールではなく、生きた存在としての比喩。

**Flow** — 情報の流れ、GTD のフロー、そして没入体験としての Flow 状態。
Anima Flow は情報の流れを知覚し、整理し、ユーザーを Flow 状態へと導く。

また、**観測→内在化→干渉**の認知ループが絶えず流れ続ける（= Flow）ことも、この名前に込められている。

## 従来の Second Brain との違い

|                | 従来の Second Brain | Anima Flow                                 |
| -------------- | ------------------- | ------------------------------------------ |
| 整理           | 人間が手動で分類    | エージェントが自律的に整理・Edge 形成      |
| 記憶           | フラットなフォルダ  | 神経科学的な記憶階層 + 連想グラフ          |
| 想起           | 人間が検索して探す  | エージェントが文脈に応じて自動想起         |
| 同期           | 人間がメモを書く    | Connector が自律的に観測・Cortex に内在化  |
| 行動           | 人間がノートを見て判断 | **エージェントが判断し、Gateway 経由で現実世界に干渉する** |

最後の行が決定的な違いである。
従来の Second Brain は「見に行くもの」だが、Anima Flow は**自ら考えて働きかけてくる**。
さらに、ユーザーからの FB がループとして Cortex に戻り、エージェントが成長し続ける。

## 解決する課題

- **情報の分散**: Slack、Gmail、Linear、Todoist 等に情報が散らばり、全体像が把握できない → **観測が Cortex に内在化する**ことで統合
- **Inbox 処理の欠如**: GTD における Inbox → 整理 → 実行のフローが確立されていない → **認知ループが自動的にフローを回す**
- **タスクの曖昧さ**: 大きく曖昧なタスクが着手の障壁になっている → **内在化した文脈を使って具体的なアクションに分解**
- **知識の散逸**: 学んだこと・決めたことが記録されず失われていく → **Cortex が記憶の定着と正規化を自律的に行う**
